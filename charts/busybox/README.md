# busybox Helm chart

## Overview

This chart installs a small BusyBox Pod with a mounted PersistentVolumeClaim. It is intended for homelab, lab, and learning use: a simple way to bring up a long-running BusyBox container and verify basic Kubernetes storage behavior.

This chart is not production-ready without review. It creates a bare Pod rather than a Deployment, has no Service, and does not apply many of the common operational settings that appear in the default `values.yaml`.

- Chart name: `busybox`
- Chart version: `0.1.4`
- App version: `1.37.0`
- Chart type: `application`

## What This Chart Deploys

The chart currently renders these Kubernetes resources:

- A `PersistentVolumeClaim` named from `pvc.name`, defaulting to `test-pvc`.
- A single `Pod` named from the Helm release fullname helper.

The Pod runs:

- Container image: `busybox:1.37.0` by default.
- Command: `sleep 3600`.
- PVC mount: the `storage` volume is mounted at `pvc.mountPath`, defaulting to `/mnt/data`.

There is no Deployment, StatefulSet, DaemonSet, Service, Ingress, ServiceAccount, HPA, ConfigMap, Secret, or NetworkPolicy rendered by the current templates.

## Requirements

- Helm 3.
- A Kubernetes cluster.
- A storage class that can satisfy the configured PVC. The default is `local-path`, which is commonly used in homelab clusters such as k3s.

## Installation

Render the chart locally:

```sh
helm template busybox-demo .
```

Lint the chart:

```sh
helm lint .
```

Install the chart:

```sh
helm install busybox-demo .
```

Install with a different storage class or size:

```sh
helm install busybox-demo . \
  --set persistent.storageclass=local-path \
  --set persistent.size=2Gi
```

Upgrade an existing release:

```sh
helm upgrade busybox-demo .
```

Uninstall the release:

```sh
helm uninstall busybox-demo
```

Check the Pod and PVC:

```sh
kubectl get pod,pvc
kubectl describe pod busybox-demo
kubectl describe pvc test-pvc
```

Open a shell in the Pod:

```sh
kubectl exec -it busybox-demo -- sh
```

## Configuration

Only the values below are used by the current templates.

| Value | Default | Description |
| --- | --- | --- |
| `image.repository` | `busybox` | Container image repository. |
| `image.tag` | `1.37.0` | Container image tag. |
| `image.pullPolicy` | `IfNotPresent` | Image pull policy for the BusyBox container. |
| `nameOverride` | `""` | Overrides the chart name used by helper templates. |
| `fullnameOverride` | `""` | Overrides the full Pod name generated by helper templates. |
| `pvc.name` | `test-pvc` | Name of the PersistentVolumeClaim and the claim mounted by the Pod. |
| `pvc.mountPath` | `/mnt/data` | Mount path for the PVC inside the BusyBox container. |
| `persistent.storageclass` | `local-path` | StorageClass name used by the PVC. |
| `persistent.size` | `1Gi` | Requested PVC size. |

`values.yaml` also contains common scaffolded settings such as `replicaCount`, `service`, `ingress`, `resources`, probes, autoscaling, extra volumes, `nodeSelector`, `tolerations`, and `affinity`. These are not currently referenced by the chart templates, so changing them will not affect the rendered Kubernetes resources.

## Persistence

Persistence is always enabled in the current chart. The PVC is created with:

- Access mode: `ReadWriteOnce`.
- StorageClass: `persistent.storageclass`.
- Size: `persistent.size`.

The Pod mounts the PVC named by `pvc.name` at `pvc.mountPath`.

Helm will delete the PVC object when the release is uninstalled, unless the cluster or storage provisioner retains the underlying volume data by policy. Review your StorageClass reclaim policy if you care about keeping the data after uninstalling the release.

## Secrets

The current chart does not create or consume Kubernetes Secrets. It does not configure passwords, environment variables, or Secret-backed volumes.

`imagePullSecrets` exists in `values.yaml`, but it is not currently used by the Pod template.

## Services and Access

The chart does not create a Kubernetes Service or Ingress. The BusyBox Pod is mainly accessed with `kubectl exec`.

Example:

```sh
kubectl exec -it busybox-demo -- sh
```

Because no Service is created, there is no cluster DNS name, NodePort, LoadBalancer, or HTTP endpoint provided by this chart.

## Helm Tests

No Helm test hooks are included in this chart. `helm test busybox-demo` will not run any chart-provided tests unless tests are added later.

## Troubleshooting

### Pod is stuck in `Pending`

Likely causes:

- The configured StorageClass does not exist.
- The cluster cannot provision a volume for the PVC.
- The node cannot satisfy the PVC access mode or local storage requirements.

Useful checks:

```sh
kubectl describe pod busybox-demo
kubectl describe pvc test-pvc
kubectl get storageclass
```

### PVC is stuck in `Pending`

Likely causes:

- `persistent.storageclass` is set to a StorageClass that is not installed.
- The storage provisioner is unavailable or misconfigured.
- The requested size is not supported by the storage backend.

Try setting the storage class to one that exists in your cluster:

```sh
helm upgrade busybox-demo . --set persistent.storageclass=<your-storage-class>
```

### Image pull errors

Likely causes:

- The node cannot pull from the configured image registry.
- `image.repository` or `image.tag` is incorrect.
- A private image was configured, but the chart does not wire `imagePullSecrets` into the Pod template.

Check the Pod events:

```sh
kubectl describe pod busybox-demo
```

### `kubectl exec` fails because the Pod completed

The Pod runs `sleep 3600`, so it exits after roughly one hour. Since this chart creates a bare Pod, Kubernetes will not recreate it like a Deployment would.

Reinstall or upgrade the release to create the Pod again:

```sh
helm upgrade --install busybox-demo .
```

## Limitations

- Homelab and learning chart only; review carefully before any production use.
- Creates a bare Pod, not a Deployment or StatefulSet.
- No Service or Ingress is created.
- No liveness, readiness, or startup probes are rendered.
- No resource requests or limits are rendered.
- No ServiceAccount is rendered, despite `serviceAccount` values existing in `values.yaml`.
- No scheduling controls are rendered, despite `nodeSelector`, `tolerations`, and `affinity` values existing in `values.yaml`.
- No Secret support is rendered.
- The Pod command is fixed to `sleep 3600`.
- Persistence is always on and there is no value to disable PVC creation.
